# Dataset Card: `products`

Following the spirit of [Datasheets for Datasets](https://arxiv.org/abs/1803.09010) (Gebru et al., 2018).

## Summary

A fully synthetic catalog of 312 products across 10 categories (Electronics, Home & Kitchen, Books, Clothing, Sports & Outdoors, Beauty, Toys, Grocery, Office Supplies, Pet Supplies), generated by combinatorially expanding hand-written vocabulary (nouns, adjectives, materials, description templates) with a seeded random number generator. **No real brand names, product listings, reviews, or scraped content are used anywhere in this dataset.**

It exists so GlassCart's search, ranking, and recommendation subsystems have a realistic-shaped corpus to index and query, without depending on any external API, license, or scraped marketplace data.

## Schema

| Field           | Type     | Description                                              |
|-----------------|----------|------------------------------------------------------------|
| `id`            | string   | Stable product id, e.g. `P00042`                          |
| `title`         | string   | `{Adjective} {Noun} — {Material}`                          |
| `description`   | string   | One templated sentence combining adjective/noun/material   |
| `category`      | string   | One of the 10 top-level categories                          |
| `brand`         | string   | Synthetic brand name (first word of a synthetic seller name)|
| `seller_id`     | string   | Id of the synthetic seller/vendor                            |
| `price`         | float    | USD, sampled per-category from a realistic range            |
| `currency`      | string   | Always `"USD"`                                              |
| `rating`        | float    | 1.0–5.0, sampled from a triangular distribution skewed high |
| `rating_count`  | int      | 0–4200, uniform                                              |
| `tags`          | string[] | 2–3 category-relevant tags                                    |
| `material`      | string   | Construction material used in the title/description          |
| `created_at`    | string   | ISO date, 2023–2026                                          |

## Provenance & Generation Process

Generated entirely by [`generate.py`](generate.py) with a fixed seed (`20260704`), so **the dataset is byte-for-byte reproducible**:

```bash
uv run datasets/products/generate.py
```

The generator:
1. Defines a small vocabulary per category (nouns, adjectives, materials, tags).
2. For each noun, samples up to 3 adjective/material variants — this intentionally creates near-duplicate products (same noun, different modifiers), which is realistic for a marketplace and gives the search/ranking subsystems something non-trivial to disambiguate.
3. Fills a description template and assigns price/rating/seller deterministically from the seeded RNG.

No network access, no external API keys, and no manual curation are required.

## Product Images

Products themselves are synthetic (see above), so there is no real photography to source per SKU. [`fetch_stock_photos.py`](fetch_stock_photos.py) instead fetches **one real, permissively-licensed stock photo per product noun** (e.g. one photo for every "Wireless Mouse" variant, regardless of adjective/material) from [Wikimedia Commons](https://commons.wikimedia.org) — a public, keyless, rate-limit-friendly JSON API, no scraping, no paid service:

```bash
uv run datasets/products/fetch_stock_photos.py
```

Only CC0/CC-BY/CC-BY-SA/public-domain images are used, and every download is recorded in `images/ATTRIBUTION.json` (commons title, artist, license, license URL, which product ids reuse it) — synced to the web app and surfaced per-image via the Glass Mode "PHOTO" badge, since attribution is a license requirement, not a nicety.

**This matching problem is genuinely hard and the pipeline is honest about its hit rate.** A short, generic noun ("Blender," "Spice Set") against a general-purpose media archive collides constantly with unrelated content. Six escalating rounds of filtering, query-tuning, and manual review were needed before settling on the current result:

1. A first pass ranked Commons' top-10 results by naive keyword overlap and shipped straight through — this produced actively wrong matches (a 19th-century painting for "Card Game," a WWII grenade-production photo for "Sticky Note Pack," both sharing one incidental word with the noun).
2. A second pass stopped re-ranking (trusting Commons' own relevance order instead), required *every* significant noun word to appear in the candidate title, and added a blocklist of historical/military/document keywords plus a historical-year regex. This cut the false-positive rate substantially but a full manual review of every accepted match still found 11 of 25 (44%) were wrong in ways no keyword filter caught — a 3D software render for "Blender" (the *tool*, not the appliance), a diagram from inside a manual for "Reference Manual," a bag embroidered "HAIR DRYER" for "Hair Dryer," an antique museum piece for "Spice Set," and others documented in [`fetch_stock_photos.py`](fetch_stock_photos.py)'s `REJECTED_COMMONS_TITLES`.
3. A push for higher coverage widened the candidate pool per query (10 → 25 results) and added ~83 hand-picked `QUERY_OVERRIDES` — more literal or disambiguating search phrases for nouns whose bare noun is either a collision risk ("Blender" → "blender jug kitchen") or too generic to find a specific match on its own. This surfaced dozens more candidates, but manual review rejected roughly two-thirds of them: further museum/archive pieces, a busy candid street-fashion photo with third-party billboards in frame, a cluttered gym-closet interior where the actual product was barely visible, and — notably — an AI-generated (DALL·E) image of an anthropomorphic rhinoceros in a business suit that Commons hosts under a "public domain" license. That last one directly contradicts this project's decision not to ship AI-generated product imagery (see "Built and evaluated, not shipped" in `docs/roadmap.md`), so `NEGATIVE_KEYWORDS` now also blocks common AI-generation markers ("dall-e," "midjourney," "stable diffusion," etc.).
4. A few Commons collections/photographers turned up as *repeat* offenders — the same wrong match resurfacing under a new filename each time a previous bad title was denylisted (an avant-garde runway denim-jacket archive, a digitized 19th-century artist's sketchbook, a nature-photography series, a historic-house-museum exhibit). Denylisting these by phrase (e.g. blocking any candidate whose title contains "Yale Center for British Art") proved more robust than denylisting one accession number at a time.
5. Nouns with no acceptable candidate across every attempt are simply skipped — including "Reference Manual" and "Automatic Feeder," where Commons' results are consistently diagrams or industrial/aquaculture equipment, never a consumer product, no matter the query phrasing.

The result: **23 of 104 nouns (~22%) have a real, hand-verified photo; the remaining 81 (~78%) fall back to the procedural gradient placeholder** (see `apps/web/src/components/ProductImage.tsx`) exactly as they did before this pipeline existed. This ceiling isn't a tuning failure — it reflects that Wikimedia Commons, an encyclopedic media archive, simply doesn't contain a good photo for most short, generic marketplace nouns at any query phrasing tried, and a wider net widens the *false*-positive supply at least as fast as the true-positive one. A low hit rate that's been visually verified beats a high hit rate that hasn't — this dataset would rather under-illustrate the catalog than misrepresent a product.

## Intended Use

- Indexing corpus for the search subsystem (BM25 + semantic embeddings).
- Candidate pool for the recommendations subsystem (content-based "similar products," reusing the search embeddings — see [docs/subsystems/recommendations.md](../../docs/subsystems/recommendations.md)), and for ranking/pricing subsystems (future work).
- Teaching example for how to structure a synthetic-data generator that is transparent about being synthetic (see [Glass Mode](../../docs/glass-mode.md)).

## Non-Intended Use

- Not intended to represent real market prices, real product availability, or real brand reputations. Do not use this dataset to draw conclusions about any real company, product, or market.
- Not a benchmark dataset for academic IR/recsys research — it is a small, templated corpus optimized for teaching and local development, not for measuring state-of-the-art retrieval quality.

## Limitations & Bias Discussion

- **Templated language**: descriptions follow 4 fixed sentence templates, so lexical diversity is much lower than a real marketplace corpus — keyword search will look artificially easy compared to production data.
- **Category vocabulary is hand-authored** by the GlassCart maintainers and reflects their (limited, English-language, US-centric) assumptions about each category's common products — it is not sourced from any market survey.
- **Rating distribution is synthetic and skewed high** (triangular, mode 4.4) to mirror the well-known positive skew in real e-commerce ratings, but the skew parameters are a guess, not fit to real data.
- **Images cover ~22% of nouns, one shared photo per noun**: see "Product Images" above. Even where a photo exists, it's a real product of that *type* — never the specific synthetic SKU — since the catalog is combinatorial and no real photo corresponds to e.g. "Ultra-Slim Smart Watch — Carbon Fiber" specifically. Any vision-related subsystem (OCR, tagging, duplicate detection) would need a purpose-built image source before it could use this dataset meaningfully. A separate CPU-only diffusion pipeline for *generating* placeholder images was also built and benchmarked (see [`training/product_images/`](../../training/product_images/) and the [model card](../../models/product-images/MODEL_CARD.md)) but its output wasn't judged good enough to ship; it produces no images checked into this repo.

## Regeneration / Extension Instructions

To change the size or shape of the catalog, edit `CATEGORIES` in [`generate.py`](generate.py) (add nouns/adjectives/materials or whole new categories) and re-run the script — every downstream artifact (embeddings, BM25 index, model cards referencing the corpus size) is derived from this file and should be regenerated afterward via:

```bash
uv run datasets/products/generate.py
uv run training/search_embeddings/build_index.py
```
